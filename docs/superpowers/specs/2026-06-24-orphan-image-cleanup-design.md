# 清除孤兒圖片功能 — 設計

狀態：**已定案，待實作**。本文件記錄 2026-06-24 與使用者討論後確認的設計,供撰寫實作計畫使用。

---

## 一、需求背景

圖片實體檔放在 Cloudflare R2(透過 S3 介面),上傳流程為「前端跟後端要臨時上傳網址 → 直接 PUT 到 R2 → 呼叫 confirm 把圖片資料寫入資料庫」。

R2 上會累積「沒有任何資料庫資料指向、卻仍佔空間」的孤兒檔。經查程式碼,實際的孤兒來源為:

1. **刪除代購上架商品 / 連線公告時**:`listing.ts` 的 delete(第 240-241 行)只刪資料庫的 `listing_images` 與 `listings` 資料列,R2 上的圖完全沒刪;`connection.ts` 的 delete 同樣。這是最大宗的孤兒來源。
2. **代購 / 連線換圖時**:換圖採「先刪舊資料列、再寫新資料列」(`replace_listing_images` / `replace_connection_images` RPC),R2 上的舊檔沒刪。
3. **換大頭貼**:`profiles` / `sellers` 的 `avatar_url` 被覆蓋,舊頭像留在 R2。
4. **送出當下 R2 已上傳但資料庫 confirm 失敗**:商品建立流程有補償刪除(`use-deferred-product-create.ts` 第 76 行會回頭刪 R2),代購 / 連線不一定有。

**非孤兒來源(已確認):** 商品、代購、連線、大頭貼、橫幅表單皆採「延遲上傳」(`pendingFiles`),圖片在按送出時才傳 R2,因此「填一半放棄表單」基本上不會留孤兒。橫幅刪除時已有刪 R2(`banner.ts` 第 107 行)。

---

## 二、已確認的決策

1. **觸發方式**:管理員在後台**手動按鈕**(不是自動排程)。
2. **流程**:掃描 → **先列出待刪清單供管理員確認** → 確認後才刪。
3. **涵蓋範圍**:全部六種用途(商品、代購、連線、大頭貼、訊息附圖、橫幅)。
4. **安全門檻**:只刪「上傳**超過 24 小時**」的孤兒,避開「正在送出、R2 已上傳但資料庫尚未寫入」的空窗,以及大頭貼即時上傳路徑。
5. **清單呈現**:**不分頁**(數量少,一次列完);**要顯示每張的用途分類**(商品 / 代購 / 連線 / 大頭貼 / 訊息 / 橫幅)。
6. **同時做 A + C**:A 清掉已累積的孤兒並作為長期兜底;C 從源頭止血,減少孤兒產生。

---

## 三、架構總覽

兩塊互補:

- **A(掃描清理)**:後台按鈕觸發的「標記-清除」,比對 R2 實體檔與資料庫所有被引用的 key,找出孤兒並在管理員確認後刪除。任何來源的孤兒最終都會被 A 掃到,是長期兜底。
- **C(源頭止血)**:在「圖片為該物件獨佔、刪除絕對安全」的程式路徑上,刪資料列時順手刪 R2,減少孤兒產生。

關鍵互補關係:**C 即使漏刪或失敗也沒關係,A 下次掃描會清掉**。因此 C 只需處理最安全、最高價值的情況,不必滴水不漏。

---

## 四、A:後台掃描清理

### 4.1 資料庫被引用的圖片 key 來源

掃描時需把以下所有「有被使用的 key」撈成一個集合:

| 來源資料表 | 欄位 | 形式 |
|------|------|------|
| `product_images` | `r2_key`、`thumbnail_r2_key` | 直接是 key |
| `listing_images` | `r2_key`、`thumbnail_r2_key` | 直接是 key |
| `connection_images` | `r2_key`、`thumbnail_r2_key` | 直接是 key |
| `banners` | `image_r2_key` | 直接是 key |
| `profiles` | `avatar_url` | 網址,需轉回 key |
| `sellers` | `avatar_url` | 網址,需轉回 key |
| `messages` | `image_url`、`context_image_url` | 網址,需轉回 key |

**網址轉 key**:去掉開頭的 `${R2_PUBLIC_URL}/`,剩下的即為 key。轉換時須處理 null / 空字串,以及非 R2 網址(理論上不應出現,保險起見略過)。

### 4.2 掃描流程(唯讀)

1. 列出 R2 `images/` 前綴下所有物件(`ListObjectsV2`,以 `ContinuationToken` 分頁撈完),記錄每個物件的 `Key` 與 `LastModified`。
2. 依 4.1 撈齊資料庫被引用的 key 集合。
3. 孤兒判定:`R2 物件的 Key 不在引用集合中` **且** `LastModified 距今超過 24 小時`。
4. 每個孤兒的用途分類:從 key 的路徑推斷(`images/{purpose}/...`,banner 為 `images/banner/...`)。
5. 回傳:孤兒清單(key、用途、大小、上傳時間)、總數量、可釋放總容量。

### 4.3 刪除流程(寫入)

1. 管理員在清單頁按「確認刪除」。
2. 後端**重新執行一次 4.1~4.3 的比對**(不信任前端傳回的清單),得到當下的孤兒集合。
3. 取「前端送回要刪的 key」與「後端重新算出的孤兒集合」之交集,只刪交集部分(避免掃描到確認之間有圖被重新引用而誤刪)。
4. 用現有 `deleteR2Objects` 刪除,回傳實際刪除數量。

### 4.4 後端介面

- 新增管理員專用 tRPC 程序(沿用現有 admin 權限守門),例如 `admin.scanOrphanImages`(query)與 `admin.deleteOrphanImages`(mutation)。
- 列舉 R2 需用到 `ListObjectsV2Command`(目前 `lib/r2.ts` 只有 `S3Client` 與 `deleteR2Objects`,需補列舉能力)。

### 4.5 前台介面

- 在現有 admin 後台新增一頁(例如「儲存空間清理」)。
- 進頁或按「掃描」後顯示:孤兒總數、可釋放容量,以及**不分頁**的清單,每筆顯示縮圖預覽、**用途分類**、大小、上傳時間。
- 「確認刪除」按鈕,刪除後重新整理清單。

## 五、C:源頭止血

**原則:只在「圖片為該物件獨佔、刪除絕對安全」處動手。** R2 刪除一律 best-effort(沿用 `deleteR2Objects`,個別失敗只記 log、不阻擋使用者操作)。

### 5.1 要處理的四個路徑

1. **刪代購上架商品**(`listing.ts` delete):刪 `listing_images` 資料列前,先撈出該 listing 所有 `r2_key` 與 `thumbnail_r2_key`;資料列刪除成功後再刪 R2。
2. **刪連線公告**(`connection.ts` delete):同上,撈 `connection_images` 的 key 後刪 R2。
3. **代購換圖**(`upload.ts` `confirmListingImages`):換圖前先記下舊的 key 集合,換圖成功後計算「被移除的 key = 舊 − 新」,只刪這些。
4. **連線換圖**(`upload.ts` `confirmConnectionImages`):同上。

### 5.2 刻意不在源頭處理(交給 A 兜底)

- **大頭貼**:同一張可能同時被 `profiles.avatar_url`、`sellers.avatar_url`、`auth` 的 user_metadata 指向,源頭刪有誤刪風險。由 A 檢查所有引用後清除。
- **商品圖**:商品為軟刪除(`is_removed` 標記,資料列保留)且為共用目錄,源頭判斷複雜。由 A 兜底。
- **橫幅**:刪除時已刪 R2(`banner.ts`),不動。

## 六、安全與邊界

- **掃描與刪除分離**:掃描全唯讀,只有管理員按確認才刪。
- **刪除前重新比對**:不信任前端清單,後端重算孤兒並取交集,避免競態誤刪。
- **24 小時門檻**:同時保護「送出空窗」與「即時上傳」路徑。
- **權限**:兩支程序皆限管理員。
- **best-effort 刪除**:C 與 A 的 R2 刪除個別失敗都不擋流程,漏掉的由下次 A 掃描清除。
- **網址轉 key 的健壯性**:處理 null、空字串、非 R2 網址。

---

## 七、測試重點

- 引用集合涵蓋全部七個欄位來源,且網址正確轉成 key。
- 24 小時門檻:剛上傳(未滿 24h)的孤兒不被列入。
- 刪除前重新比對:掃描後、確認前被重新引用的 key 不會被刪。
- C 各路徑:刪代購 / 連線後對應 R2 被刪;換圖後僅「被移除」的舊 key 被刪,留用的不受影響。
- best-effort:R2 刪除失敗不影響資料庫操作與使用者流程。
